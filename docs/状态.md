# 状态模型

EUI-NEO 使用显式状态和单向数据流。可以把它理解成：

> `compose()` 每次生成一张新的界面施工图；状态保存在施工图之外，下一次 compose 再把当前状态写回组件。

组件通常不拥有业务真相。页面或上层业务对象传入当前值，组件交互时通过 `onChange` 等回调给出新值，页面接受或修正后，下一次 compose 再把结果传回组件。

当前实现不是 React/Vue 式自动响应系统，也没有内置全局 Store。状态变了不一定自动刷新 UI；状态放在哪里、由谁拥有、如何触发 compose，需要开发者明确选择。

## 先看结论：什么时候用什么

```text
只在本次 compose 中使用             -> 局部变量
当前页面需要在多次 compose 间记住   -> ui.state<PageState>("page")
组件自己的光标、草稿、动画辅助状态   -> ui.state<ComponentState>(componentId)
组件当前值和变更回调                 -> 受控组件 value/open/checked + onChange
想减少受控组件绑定样板代码           -> Signal<T> + bind(...)
hover、pressed、滚动、拖动、动画     -> Runtime 状态和 Runtime binding
跨页面、跨窗口或独立业务生命周期     -> app/domain 对象
需要程序重启后恢复                   -> app/domain 对象 + 显式持久化
后台线程产生新数据                   -> 回到 UI 线程，或同步后 requestUpdate()
```

更完整的决策表：

| 场景 | 推荐方案 | 不推荐 |
| --- | --- | --- |
| 本轮布局计算、颜色混合、临时数组 | compose 局部变量 | `ui.state`、全局变量 |
| 单个页面的选中项、开关、筛选条件 | `ui.state<PageState>("page")` | 文件级可变全局变量 |
| 输入框光标、拖拽草稿、picker draft | 组件内部 `ui.state<T>(id)` | 页面替组件管理所有细节 |
| slider、checkbox、dialog 的业务值 | 受控属性 + `onChange` | 组件内部复制一份业务真相 |
| 简单双向绑定 | `Signal<T>` + `.bind(...)` | 把 `Signal` 当自动响应式 Store |
| 高频 hover、pointer-follow、滚动 | Runtime binding | 每次移动都修改页面状态并 compose |
| 多页面共享的登录用户、文档模型 | app/domain 对象 | `ui.state`、组件静态变量 |
| 配置落盘、会话恢复 | 业务对象 + 序列化 | 期待 `ui.state` 自动持久化 |
| 后台任务结果 | UI 主线程回调修改状态 | 多线程无同步读写 `ui.state` |

## 状态分层

| 层级 | 归属 | 主要 API | 适用场景 |
| --- | --- | --- | --- |
| 页面实例状态 | 当前 `Ui` / loader scope | `ui.state<T>(id)` | 页面选中项、开关、筛选条件、轻量缓存 |
| 业务状态 | 页面对象、app、domain model | 普通 C++ 对象、结构体、`Signal<T>` | 跨组件或跨页面数据、业务规则、持久化数据 |
| 组件私有状态 | 当前 `Ui` / loader scope | `ui.state<T>(id)` | 光标、拖拽草稿、picker draft、动画辅助状态 |
| Runtime 交互状态 | Runtime 实例表 | `scrollState(...)`、`sliderState(...)`、`visualStateFrom(...)` | 滚动、拖动、hover、pressed、pointer transform |
| 渲染缓存状态 | Runtime / Render backend | primitive、text、image、framebuffer cache | 脏区重绘和 GPU/CPU 资源复用 |

依赖方向保持固定：

```text
页面 / app / domain 状态
          |
          v
      受控组件 DSL
          |
          v
 Runtime 交互和动画状态
          |
          v
       渲染缓存
```

底层 Runtime 和渲染后端不应反向拥有页面业务真相。

## compose 和状态是什么关系

事件、定时器、文本输入、部分组件回调或显式更新请求会让 Runtime 请求重新 compose。compose 会清空并重新生成 DSL 元素树，但不会自动清空 `StateStore`：

```cpp
void compose(eui::Ui& ui, const eui::Screen& screen) {
    // 根据当前状态重新声明页面。
}
```

这意味着：

- DSL builder 和 compose 局部变量是短生命周期对象。
- `ui.state<T>(id)` 保存在当前 `Ui` 的 `StateStore` 中，可以跨多次 compose 存活。
- Runtime 的滚动、hover、pressed、transition 等状态保存在 Runtime 实例表中。
- 页面重新 compose 不等于重新创建窗口，也不等于全部状态归零。

## 局部变量

compose 内的普通局部变量每次都会重新创建：

```cpp
void compose(eui::Ui& ui, const eui::Screen& screen) {
    const float contentWidth = std::max(0.0f, screen.width - 48.0f);
    const auto theme = components::theme::dark();
}
```

适合保存：

- 当前窗口尺寸推导值。
- 本轮布局坐标和颜色。
- 从持久状态计算出的只读派生值。
- 只在当前 compose 调用链内使用的临时容器。

不适合保存需要被回调修改并跨 compose 保留的值：

```cpp
void compose(eui::Ui& ui, const eui::Screen&) {
    int count = 0; // 下一次 compose 又会变成 0。
}
```

## 页面实例状态：ui.state

单文件页面不需要依赖可变全局变量。推荐把相关字段放入一个页面状态结构体：

```cpp
struct PageState {
    int selectedIndex = 0;
    bool autoPlay = true;
    float animationSpeed = 0.55f;
};

void compose(eui::Ui& ui, const eui::Screen& screen) {
    PageState* state = &ui.state<PageState>("page");

    components::button(ui, "next")
        .onClick([state] {
            ++state->selectedIndex;
        })
        .build();
}
```

`ui.state<T>(id)` 的行为：

- 第一次访问时默认构造 `T`。
- 后续使用相同作用域、相同 id、相同类型访问时返回同一个对象。
- 状态 key 会包含当前 page id 和 loader scope。
- page id 改变后会访问另一组状态；旧 page id 的状态不会仅因为切页而立即自动删除。
- 同一个文本 id 使用不同类型时是不同状态项，但不建议故意依赖这个特性。
- StateStore 内的对象地址在正常 compose 期间保持稳定。
- Runtime 关闭、`clearState()` 或对应 loader scope 被释放后，对象会销毁。

### 为什么回调建议捕获指针

不要捕获 compose 栈上引用别名，也不要捕获临时 builder 的 `this`：

```cpp
PageState* state = &ui.state<PageState>("page");

components::button(ui, "next")
    .onClick([state] {
        ++state->selectedIndex;
    })
    .build();
```

这里按值捕获的是 StateStore 中对象的指针，而不是 compose 局部变量本身。该指针只在所属 `Ui` / loader scope 存活期间有效；不要把它保存到生命周期更长的全局对象、后台任务或另一个窗口中。

### 稳定 id 规则

状态 id 决定状态身份：

```cpp
ui.state<PageState>("page");
ui.state<DragState>(id + ".drag");
```

必须遵守：

- id 在同一个逻辑实例的生命周期内保持稳定。
- 列表项状态应使用稳定业务 key，不要只使用会变化的数组下标。
- 不要每次 compose 生成随机 id 或时间戳 id。
- 页面状态优先使用清楚的固定 id，例如 `"page"`、`"editor"`。
- 组件私有状态使用组件根 id 或 `id + ".state"`。

id 变化会创建新状态；旧状态不会因为本轮没有访问就自动回收，除非 scope 被释放或整个 Runtime 关闭。

## Loader 生命周期

Loader 会把内部元素和状态放入独立 scope：

```cpp
ui.loader("settings")
    .active(showSettings)
    .destroyOnHide()
    .content([&] {
        SettingsState& state = ui.state<SettingsState>("page");
    });
```

- `destroyOnHide()`：loader 关闭时释放该 scope 下的 `ui.state`。
- `keepAlive()`：loader 关闭时保留状态，下次打开继续使用。
- Runtime 关闭：清空所有 `Ui` 状态。

Loader 默认模式是 `destroyOnHide()`；只有确实需要保留隐藏页面状态时才显式使用 `keepAlive()`。

选择原则：

- 临时弹层、可重新初始化页面使用 `destroyOnHide()`。
- 返回后需要保留输入内容、滚动位置或草稿时使用 `keepAlive()`。
- 不要让 loader 内状态指针逃逸到比 loader 更长寿的对象中。

## 组件私有状态

组件可以用 `ui.state<T>(id)` 保存用户看不到或不应由页面管理的交互细节：

```cpp
struct DragState {
    bool dragging = false;
    float startX = 0.0f;
};

DragState* state = &ui.state<DragState>(id + ".drag");
```

适合：

- 输入框光标、选择区间和 IME 草稿。
- 拖拽起点和 picker 临时值。
- 组件动画进度或初始化标志。
- 解析结果、测量结果等与组件实例绑定的轻量缓存。

不适合：

- 当前登录用户、业务文档、订单等跨页面数据。
- 需要落盘恢复的数据。
- 需要后台线程长期访问的数据。
- 多个独立组件都要修改的共享业务真相。

不要使用组件级 `static unordered_map<id, state>`；它无法正确跟随窗口、页面和 loader 生命周期。

## 受控组件和单向数据流

通用组件默认是受控组件：页面传入当前值，组件通过回调请求更新。

```cpp
PageState* state = &ui.state<PageState>("page");

components::slider(ui, "volume")
    .value(state->volume)
    .onChange([state](float next) {
        state->volume = std::clamp(next, 0.0f, 1.0f);
    })
    .build();
```

数据流是：

```text
state->volume
    -> slider.value(...)
    -> 用户操作
    -> onChange(next)
    -> 页面更新 state->volume
    -> 下一次 compose 再传回 slider
```

`onChange` 的含义是“组件建议的新值”，不是“组件已经拥有了新的业务状态”。页面可以接受、校验、修正或忽略它。

标准命名：

| 状态语义 | 当前值 API | 变更 API | 示例 |
| --- | --- | --- | --- |
| 通用值 | `value(T)` | `onChange(T)` | input、slider、stepper、colorPicker |
| 布尔选择 | `checked(bool)` | `onChange(bool)` | checkbox、switch |
| 选中项 | `selected(...)` | `onChange(...)` | tabs、segmented、radio、dropdown |
| 连续位置 | `index(float)` | `onChange(float)` | carousel |
| 弹层开关 | `open(bool)` | `onOpenChange(bool)` | dropdown、dialog、sidebar |
| 短暂可见性 | `visible(bool)` | dismiss 回调 | toast |
| 滚动位置 | `offset(float)` | `onChange(float)` | scrollView、virtualList |

新组件优先沿用这些名称。不要新增 `onValueChanged`、`onSelectedChanged`、`onCheckedChanged` 等重复命名。

## Signal<T>

`Signal<T>` 当前只是一个显式值容器，也是受控组件 `.bind(...)` 的语法糖：

```cpp
struct PageState {
    eui::Signal<float> volume{0.62f};
};

PageState* state = &ui.state<PageState>("page");

components::slider(ui, "volume")
    .bind(state->volume)
    .build();
```

大致等价于：

```cpp
.value(state->volume.get())
.onChange([state](float next) {
    state->volume.set(next);
})
```

常用绑定：

| 组件状态 | Signal API | 等价受控写法 |
| --- | --- | --- |
| `value(...)` | `bind(signal)` | `value(signal.get()).onChange(...)` |
| `checked(...)` | `bind(signal)` | `checked(signal.get()).onChange(...)` |
| `selected(...)` | `bind(signal)` | `selected(signal.get()).onChange(...)` |
| `index(...)` | `bind(signal)` | `index(signal.get()).onChange(...)` |
| `offset(...)` | `bind(signal)` | `offset(signal.get()).onChange(...)` |
| `open(...)` | `bindOpen(signal)` | `open(signal.get()).onOpenChange(...)` |
| `visible(...)` | `bindVisible(signal)` | `visible(signal.get()).onDismiss(...)` |

重要限制：

- `Signal::set()` 不会自动订阅依赖，也不会自行请求 compose。
- `Signal` 没有观察者列表、派生值、批处理或事务语义。
- 在 compose 中临时创建 `Signal`，下一次 compose 仍会被重新创建。
- `Signal` 不是线程安全容器。
- 不要在同一个组件上同时使用 `bind(...)` 和同类 `onChange(...)`，后设置的回调会覆盖前面的绑定。

如果回调还需要校验、日志或联动其它状态，优先写显式受控属性和回调。

## Runtime 状态和高频交互

hover、pressed、滚动、slider 拖动和 pointer-follow 等高频状态优先留在 Runtime，避免鼠标每移动一个像素就重新 compose 整棵页面：

```cpp
ui.stack("button")
    .visualStateFrom("button.surface", 0.965f)
    .build();

ui.stack("viewport")
    .scrollState("viewport", scrollY, maxOffset, 48.0f)
    .build();
```

常见 API：

- `visualStateFrom(...)`：读取另一个元素的 hover/pressed 视觉状态。
- `runtimePointerTransformFrom(...)`：由 Runtime 根据指针驱动 transform。
- `runtimePointerTiltFrom(...)`：由 Runtime 驱动倾斜效果。
- `scrollState(...)`：维护滚动交互状态。
- `sliderState(...)`：维护 slider 拖动状态。
- `scrollContentFrom(...)`、`scrollThumbFrom(...)`：从 Runtime 滚动状态派生视觉位置。

什么时候仍然使用回调状态：

- 交互会改变业务数据。
- Runtime 暂时无法表达该视觉效果。
- 需要回传拖拽结果、选择结果或业务坐标。

纯视觉 pointer-follow 优先使用 Runtime binding，不要用 `onMove` 持续写页面状态。

## app/domain 状态

以下状态应由 `Ui` 之外的 app 或 domain 对象拥有：

- 多个页面或窗口共享的数据。
- 有明确业务规则和生命周期的数据模型。
- 需要后台任务访问的数据。
- 需要保存到文件、数据库或网络的数据。
- 需要单元测试且不应依赖 UI 的业务逻辑。

示意：

```cpp
struct DocumentModel {
    std::string path;
    std::string content;
    bool dirty = false;
};

class EditorSession {
public:
    DocumentModel document;
};
```

页面只读取模型并发出操作请求。不要为了避免定义业务对象，把所有共享数据都塞进 `ui.state<PageState>`。

## 全局变量

全局变量不是完全禁止，但应只用于真正的进程级状态，例如只读配置或明确的应用单例。页面交互状态通常不应该使用文件级可变全局变量。

可变全局状态的问题：

- 多窗口和多页面会意外共享同一份值。
- 页面销毁后状态仍存在。
- 测试之间容易相互污染。
- 初始化和销毁顺序不清晰。
- 后续无法自然支持多个页面实例。
- 并发访问时容易产生数据竞争。

单文件示例也优先使用 `ui.state<PageState>("page")`，这样示例展示的就是可扩展写法。

## 状态改变后如何刷新 UI

修改变量本身不一定会自动重新 compose。刷新来源分为三类。

### UI 事件回调

点击、输入、滚轮、拖拽、timer 和部分组件事件由 Runtime 派发。相关回调执行后，Runtime 通常会设置 compose/paint 请求：

```cpp
.onClick([state] {
    state->open = !state->open;
})
```

这种情况下通常不需要手动调用 `app::requestUpdate()`。

### app::async 完成回调

`app::async` 的完成回调由框架主线消费；框架内部的网络和图片任务会显式唤醒 UI。页面任务修改状态后，应确认所在调用链会请求更新；不确定时调用 `app::requestUpdate()`。

注意状态生命周期：如果任务完成前页面或 loader scope 可能销毁，不要让异步回调长期持有裸 `ui.state` 指针；应把结果交给生命周期更长的业务对象，或者使用任务取消和页面存活检查。

### 自建后台线程或外部事件

自建线程、socket、文件监听等外部来源需要：

1. 用同步机制把结果交给 UI 线程可见的数据对象。
2. 调用 `app::requestUpdate()` 唤醒 UI。
3. 在 UI 线程 compose 或回调中读取并展示结果。

不要从后台线程无同步读写 `ui.state`、`Signal` 或 DSL 元素。

## 派生状态

能够从其它状态计算出来的值，优先在 compose 中现算，不要复制保存：

```cpp
const int itemCount = static_cast<int>(state->items.size());
const bool canSubmit = !state->name.empty() && !state->loading;
```

只有计算成本明显较高时才缓存派生结果，并同时保存足够的输入版本用于失效。否则容易出现源数据已变化、派生状态仍是旧值的问题。

## 常见错误

### 在 compose 中保存状态到局部变量

下一次 compose 会丢失。

### 捕获临时 builder 的 this

builder 在 `.build()` 调用结束后通常会销毁，运行期回调捕获它会形成悬空指针。回调只捕获所需值、稳定页面状态指针或生命周期明确的业务对象。

### 用随机 id 保存状态

每次 compose 都会创建新状态，旧状态可能一直保留到 scope 销毁。

### 把 ui.state 指针交给后台线程

StateStore 没有线程安全保证，而且页面或 loader 释放后指针会失效。

### 同时保存组件业务值和页面业务值

两个来源会逐渐不同步。组件使用受控属性，内部只保存交互草稿。

### 认为 Signal.set 会自动刷新

当前 `Signal` 没有订阅/通知能力。它必须处在已有 UI 回调或显式更新流程中。

### 为所有 hover 和动画执行 compose

高频纯视觉状态应交给 Runtime binding 和 transition。

## 当前局限性

### StateStore

- 只有按稳定 id 和类型存取，没有公开的单项查询、删除、重置或枚举 API。
- 未在本轮 compose 访问的普通状态不会自动回收；主要依靠 loader scope 释放或 Runtime shutdown。
- 改变 page id 不会主动清理旧 page id 对应的状态。
- 没有状态版本、变更通知、脏标记或调试面板。
- 没有序列化、持久化、迁移或热重载支持。
- 没有线程安全保证。
- 返回裸引用/指针，生命周期错误主要靠开发者约束避免。

### Signal

- 只是值容器，没有 observer/subscriber。
- `set()` 不会自动请求 compose。
- 没有 computed/derived signal。
- 没有批量更新、事务、去重通知或循环依赖检测。
- 没有线程安全和生命周期订阅解绑机制。

### 页面和业务状态

- 没有内置全局 Store、依赖注入或页面 ViewModel 基类。
- 没有统一的持久化和恢复协议。
- 没有 undo/redo、历史快照或 time-travel 调试。
- 多窗口共享状态需要应用自己定义 ownership 和同步方式。
- 异步任务与页面 scope 的生命周期还需要调用方手动协调。

### 更新机制

- 框架没有自动追踪 compose 读取了哪些状态。
- 普通 C++ 变量变化不会自动触发刷新。
- UI 事件一般会触发 compose，但外部状态变化需要 `app::requestUpdate()`。
- 目前缺少只重组某个组件子树的公开状态更新模型；业务状态 compose 通常从页面入口重新声明 DSL。

## 改进方案和优先级

下面是建议的渐进路线。目标是保留当前简单、显式的模型，而不是一次性引入复杂响应式框架。

### P1：补齐状态生命周期和更新基础 API

建议优先增加：

- `ui.resetState<T>(id)`：显式重置单个状态。
- `ui.releaseState(id)` / `releaseStateScope(id)` 的完整公开说明。
- Debug 模式下检测同一 id 的异常类型变化。
- Debug 状态统计：条目数量、scope、类型和长期未访问状态。
- 明确的 UI 线程断言，尽早发现后台线程误用。

价值：解决当前最容易出现的状态泄漏、错误复用和生命周期不可见问题，且不改变现有编程模型。

### P2：让 Signal 成为可选通知源

可以在保持 `Signal<T>` 简单 API 的前提下增加：

- 值变化去重。
- 可选订阅和自动解绑 token。
- 可选 `requestUpdate` 调度器绑定。
- `ComputedSignal<T>` 或轻量只读派生值。
- UI 线程批量更新，将多次 set 合并为一次 compose 请求。

必须避免：

- 隐式全局依赖图。
- 无法追踪的魔法订阅。
- Signal 自己持有页面或组件裸指针。

### P3：提供明确的页面状态宿主

当页面变复杂时，可增加轻量 Page/ViewModel 约定：

```cpp
struct MyPageState { /* UI state */ };
class MyPageModel { /* business state and actions */ };
```

框架可以提供：

- 页面进入、离开和销毁钩子。
- 异步任务自动跟随页面取消。
- 页面状态与业务模型的明确注入入口。
- 多窗口实例各自持有 PageState，共享显式传入的 Model。

不建议直接引入重量级全局 Store；先把 ownership 和生命周期做好。

### P4：持久化和历史能力

在业务模型层提供可选能力：

- 显式 serialize/deserialize。
- schema version 和迁移。
- dirty tracking。
- command/undo/redo。
- 可测试的状态快照。

这些能力不应默认塞进 `ui.state`，因为组件草稿、hover 和缓存通常不应该被持久化。

### P5：更细粒度的重组

如果性能数据证明页面级 compose 成为瓶颈，再考虑：

- loader/subtree 级 compose 请求。
- 状态读取依赖追踪。
- keyed component scope。
- 批量状态更新和局部脏标记。

在没有性能证据前，优先保留当前显式 compose + Runtime 高频状态的简单结构。

## 新代码检查清单

提交状态相关代码前检查：

- 这个值是临时计算、页面状态、组件私有状态、Runtime 状态还是业务状态？
- 状态 owner 和销毁时机是否明确？
- id 是否稳定，列表项是否使用稳定 key？
- 是否不必要地复制保存了派生状态？
- 组件是否保持受控，是否出现两份业务真相？
- 回调是否捕获临时 builder、栈引用或可能提前销毁的对象？
- 高频纯视觉变化是否可以改用 Runtime binding？
- 外部线程修改后是否正确同步并请求 UI 更新？
- loader 使用 `destroyOnHide()` 还是 `keepAlive()` 是否符合预期？
- 需要跨页面或持久化的数据是否已经提升到 app/domain 层？
