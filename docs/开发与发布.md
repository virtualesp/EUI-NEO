# 开发与发布

本文记录 EUI-NEO 的本地开发、依赖模式、构建验证和 release 约定。目标是让每次改动都能被快速复现、审查和发布。

## 依赖模式

项目通过 `EUI_DEPS_MODE` 控制构建期依赖来源：

- `auto`：默认模式。优先使用 `3rd/` 下的本地源码，缺失时才从固定上游地址联网拉取。
- `bundled`：严格离线模式。只使用 `3rd/` 下的源码，缺失依赖直接报错。
- `fetch`：强制联网模式。从固定上游地址拉取构建期依赖。

HarfBuzz shaping 默认启用，可用 `-DEUI_ENABLE_HARFBUZZ=OFF` 关闭。

内置和 fetch 下载的第三方依赖默认按静态链接构建，包括 GLFW、glad、FreeType、HarfBuzz、libpng 和 zlib。Release 包不再额外收集 GLFW DLL / dylib / so。只有选择系统 SDL2 包时，SDL2 仍可能按系统包提供的方式动态链接。

## 渲染后端选择

项目通过 `EUI_RENDER_BACKEND` 控制渲染后端：

- `auto`：默认模式。Vulkan SDK 存在时选择 Vulkan，否则回退 OpenGL。
- `opengl`：强制 OpenGL。
- `vulkan`：强制 Vulkan；如果 SDK 不存在，配置阶段直接报错。

Vulkan 后端已作为正式渲染后端接入。需要测试 Vulkan 时使用：

```bash
cmake -S . -B build/vulkan-glfw -DEUI_WINDOW_BACKEND=glfw -DEUI_RENDER_BACKEND=vulkan
cmake --build build/vulkan-glfw --target gallery --parallel
```

也可以直接使用预设测试 primitive-only 视觉验收入口：

```bash
cmake --preset vulkan-glfw-release
cmake --build --preset vulkan-glfw-release --target primitive_parity --parallel
cmake --preset vulkan-sdl2-release
cmake --build --preset vulkan-sdl2-release --target primitive_parity --parallel
```

`auto` 在检测到 Vulkan SDK 时会优先使用 Vulkan，否则回退到 OpenGL。

显式 OpenGL / Vulkan 对照测试推荐使用预设：

```bash
cmake --preset opengl-glfw-release
cmake --build --preset opengl-glfw-release --target gallery primitive_parity --parallel
cmake --preset vulkan-glfw-release
cmake --build --preset vulkan-glfw-release --target gallery primitive_parity --parallel
```

窗口后端默认是 GLFW。SDL2 不放进 `3rd/`，只有配置 `-DEUI_WINDOW_BACKEND=sdl2` 才参与构建：

- `EUI_DEPS_MODE=auto`：优先查找系统 SDL2 包，找不到时下载固定版本 SDL2。
- `EUI_DEPS_MODE=bundled`：SDL2 后端直接报错，因为没有 vendored SDL2。
- `EUI_DEPS_MODE=fetch`：下载固定版本 SDL2 并参与构建。

## CMake Targets

项目导出 `eui::neo` 静态库 target。顶层构建时 `EUI_BUILD_APPS` 默认开启，会构建 `examples/*.cpp` 下的示例程序；作为其他项目的子目录接入时，`EUI_BUILD_APPS` 默认关闭，只提供库和辅助函数。

- `eui::neo`：核心 DSL、Runtime、图元、组件可用的库 target。
- `eui_neo_configure_app(target)`：给完整 EUI app 配置平台链接选项并复制 `assets/`。
- `eui_neo_copy_assets(target)`：只复制 EUI-NEO 自带运行资源。

## 本地构建

macOS / Linux:

```sh
cmake --preset glfw-release
cmake --build --preset glfw-release
./build/glfw-release/gallery
```

`glfw-release` 使用 `EUI_RENDER_BACKEND=auto`。如果机器装有 Vulkan SDK，它会选择 Vulkan；要固定 OpenGL，请用 `opengl-glfw-release`。

Windows / PowerShell:

```powershell
cmake -S . -B build
cmake --build build --config Release
.\build\Release\gallery.exe
```

Linux 依赖示例：

```sh
sudo apt-get install -y ninja-build libx11-dev libxrandr-dev libxinerama-dev libxcursor-dev libxi-dev libgl1-mesa-dev libcurl4-openssl-dev
# SDL2 后端可选：
sudo apt-get install -y libsdl2-dev
```

SDL2 后端使用系统包：

```sh
cmake --preset sdl2-release
cmake --build --preset sdl2-release --target gallery
```

SDL2 后端显式 fetch：

```sh
cmake --preset sdl2-fetch-release
cmake --build --preset sdl2-fetch-release --target gallery
```

## 验证命令

常规改动至少验证严格离线构建：

```sh
cmake -S . -B /tmp/eui-bundled -DCMAKE_BUILD_TYPE=Release -DEUI_DEPS_MODE=bundled
cmake --build /tmp/eui-bundled --parallel
```

依赖链路改动额外验证联网 fetch：

```sh
cmake -S . -B /tmp/eui-fetch -DCMAKE_BUILD_TYPE=Release -DEUI_DEPS_MODE=fetch
cmake --build /tmp/eui-fetch --parallel
```

发布前检查源码包体积：

```sh
git archive --format=zip --output=/tmp/eui-source.zip HEAD
ls -lh /tmp/eui-source.zip
```

## 提交规范

项目提交使用带 scope 的 Conventional Commits：

```text
type(scope): subject
```

允许的 type：

- `feat`：新增功能。
- `fix`：修复问题。
- `docs`：文档改动。
- `style`：格式、空白、代码风格，不改变行为。
- `refactor`：重构，不新增功能也不修复问题。
- `perf`：性能优化。
- `test`：测试相关。
- `build`：构建系统或依赖管理。
- `ci`：CI / release workflow。
- `chore`：维护性改动。
- `revert`：回滚提交。

规则：

- 必须包含 scope，例如 `docs(readme)`、`ci(release)`、`fix(input)`。
- type 和 scope 使用小写英文。
- subject 使用简短英文祈使句，不加句号。
- 避免 `update docs`、`fix bug`、`change files` 这类模糊描述。
- 提交前用 `git log --oneline -5` 确认新提交信息符合规范。

示例：

```text
docs(readme): align dependency setup documentation
ci(release): add macos release packaging
docs(workflow): add development and review guides
fix(input): correct emoji caret positioning
feat(text): add bundled emoji rendering
chore(deps): trim bundled third-party sources
```

## Release 约定

- 推送 `v*` tag 会触发 `.github/workflows/release.yml`。
- CI 构建 Windows、Linux、macOS 运行包。
- Release assets 只上传运行包，不上传正文截图。
- Release 正文截图使用已提交的仓库图片或 GitHub 正文附件。
- `docs/releases/` 是本地 release 草稿和素材目录，已被 `.gitignore` 忽略，不提交。

## Release 检查

发布前确认：

- `git status --short` 为空。
- `vX.Y.Z` tag 指向预期提交。
- Windows、Linux、macOS workflow 均通过。
- Release assets 只包含运行包。
- GitHub 自动 Source code 包体积处于合理范围。
- Release note 中的截图能在网页正文显示。
